**Note on the package's own planning deviation:** the package's own §5 originally specified a
per-identity named-patch-key scheme (`water_mus_lo`/`water_mus_hi`, etc.), but the implementing
commit shipped a flat, biome-id-indexed `music_table` instead — because the named-key scheme
would have required `asm_game.py` changes, contradicting this package's own explicit
no-`asm_game.py`-changes scope boundary (`build_rom.py:110-122` documents this in place, and
`docs/architecture/09-interface-specification.md:83-88` records it as an architecture-level
delta). This is not a defect: the Interfaces §5 language itself was aspirational text describing
an approach that turned out to be infeasible within the package's own declared scope, and the
implementation caught the contradiction and used the already-named fallback (§5's own "flat
ROM-resident pointer array" alternative) rather than silently violating scope. This is already
tracked as an Outstanding Issue for `IP-1111`'s own future `07-implementation-planning` touch
(consume `music_table` by biome-id index, not named keys) — correctly not this package's own
problem to fix.
